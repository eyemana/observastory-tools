# Observastory 1.0 extension tutorial

Observastory 1.0 separates four concerns:

1. Folder and entity configuration determines **what exists**.
2. An evaluation contract determines **what is measured and which bounded context is allowed**.
3. A metric note supplies the author's **interpretive meaning**.
4. Reports project canonical `ai.observations`; they discover configured and observed axes automatically.

The built-in names Character, Plot Thread, Story Engine, Arc, and Narrator are tutorial defaults,
not a required theory of story. Rename, remove, or add entity types to match the project.

## Add a standard metric

Create `Metrics/Resonance.md`, describe what resonance means, add any optional execution settings
under `standardMetrics.metrics.Resonance`, and schedule pairs such as:

```json
["Resonance", "Motif"]
```

The standard evaluator writes the result under
`ai.observations.<entity type>.<entity name>.resonance`.

## Add a story-element type

Map the writer's folder without changing evaluator code:

```json
"story": {
  "folders": { "motifs": "Motifs" },
  "entityTypes": {
    "motifs": {
      "target": "motif",
      "folderKeys": ["motifs"],
      "label": "motif",
      "pluralLabel": "motifs"
    }
  }
}
```

Notes in that folder become the eligible definition set. The filename is the entity identity;
the note body is the author's definition.

## Add an observer-target relationship

Relationship configuration chooses the observer, targets, ordering context, dimension, and
primary value. This example evaluates each configured character's current trust in a narrator:

```json
"evaluation": {
  "relationships": {
    "characterTrust": {
      "metric": "Trust",
      "dimension": "trust",
      "targets": ["narrators"],
      "valueKind": "score",
      "priorContext": "chronology",
      "observer": {
        "mode": "entities",
        "key": "characters",
        "target": "character",
        "storageKey": "characters"
      },
      "meaning": "Measure the observer's warranted trust in the target.",
      "knowledgeBoundary": "Use only evidence plausibly available to that observer by this point in chronology."
    }
  }
}
```

Then schedule `["Trust", "Narrator"]`. A constant observer such as the reader uses
`"mode": "constant"` with a configured `type`, `name`, and `storageKey`.

Reader Awareness and Character Awareness are ordinary shipped instances of this contract.
They are not evaluator branches or restrictions on which relationships an author may define.

## Track change without defining what an arc must be

A trajectory is optional behavior that can target any entity type:

```json
"evaluation": {
  "trajectories": {
    "change": {
      "metric": "Trajectory",
      "dimension": "trajectory",
      "targets": ["arcs"],
      "priorContext": "readerOrder",
      "meaning": "Use the definition's own model of change. Do not assume fixed stages or forward progress."
    }
  }
}
```

The target definition can describe an ordered sequence, spectrum, accumulation, decline,
oscillation, reversal, cycle, or informal change in prose. The evaluator reports only what the
scene supports: `stateBefore`, `stateAfter`, `transition`, `movement`, `clarity`, `confidence`,
and `evidenceStrength`. Empty state labels are valid when the scene does not establish them.

The tutorial schedules this capability against `arcs`, but the engine has no arc-specific state
model. The same contract may target motifs, relationships, institutions, promises, public image,
or any custom entity type.

## Reports

`report-catalog.cjs` merges configured evaluation pairs with canonical observations already in
the manuscript. Observation Trajectory, Metric Heatmaps, Goal Heatmap, Goal Bullseye, and Scene
Profile use that catalog. A new type or dimension therefore appears after configuration or its
first observation without editing a report's entity list.

## Stable 1.0 boundaries

- Official scenes are notes under the configured Scenes folder with `type: scene`.
- Other notes under Scenes are fragments and are evaluated only through official scenes that embed them.
- Prose `![[embeds]]` expand transiently; ordinary `[[links]]` remain links and attention signals.
- Metadata embeds do not expand in 1.0.
- Chapters remain a planning/grouping projection over scenes, not an evaluation unit.
- Generated observations and indexes live under `ai`, `.index`, or `.queue`; author prose and links remain ordinary Obsidian content.

Run `npm run verify` in `observastory-tools` before treating an extension as compatible with 1.0.
